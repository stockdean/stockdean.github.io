#存储

##数据在外存的组织结构

###文件
构成MySQL数据库和InnoDB存储引擎表的各类文件：

  1. 参数文件：告诉MySQL实例启动时在哪里可以找到数据库文件，并且定某些初始化参数，这些参数定义了某种内存结构的大小等设置。
  2. 日志文件：用来记录MySQL实例对某种条件做出响应时写入的文件，如错误日志文件、二进制日志文件、慢查询日志文件、查询日志文件等。
  3. socket文件：当用UNIX域套接字方式进行连接时需要的文件。
  4. pid文件：MySQL实例的进程ID文件。
  5. MySQL表结构文件：用来存放MySQL表结构定义文件，对每张表MySQL创建一个后缀为frm的文件记录表结构定义，包括视图的定义。
  6. 存储引擎文件：每个存储引擎都会有自己的文件来保存各种数据，这些存储引擎真正存储了记录和索引等数据。
  7. 临时表空间文件：MySQL 5.7 新开辟了一个临时表空间，默认的磁盘文件命名为ibtmp1，所有非压缩的临时表都存储在该表空间中。由于临时表的本身属性，该文件在重启时会重新创建。

    InnoDB存储引擎文件

    对于每个表，存储引擎有自己独有的文件。与InnoDB相关的文件包括：表空间文件、重做日志文件（记录了对于InnoDB存储引擎的事务日志）。

    表空间文件

    InnoDB采用将存储的数据按表空间（tablespace）进行存放的设计。在默认配置下有一个初始大小为10MB、名为ibdata1的文件——默认表空间文件（tablespace file）。通过参数innodb_data_file_path对其进行设置。

可以由多个文件组成一个表空间。

设置innodb_data_file_path后，所有基于InnoDB的表的数据都会记录到该共享表空间中。

设置innodb_file_per_table，每个基于InnoDB的表产生一个后缀名为ibd的独立表空间文件，此时不需将所有数据存放于默认的表空间（共享表空间）中。

单独的表空间文件仅存储该表的数据、索引和插入缓冲Bitmap等信息，区域信息（如回滚（undo）信息、插入缓冲索引页、系统事务信息、二次写缓冲（Double write buffer）等）仍存放在默认表空间（共享表空间）中。 这部分参见：关于innodb_file_per_table的设置对表存储的影响说明

表

索引组织表

在InnoDB中，表都是根据主键（Primary Key）顺序组织存放的——索引组织表。

InnoDB逻辑存储结构

所有数据都被逻辑地存放在一个空间——表空间（tablespace）中，表空间（IDB文件）由段（segment）、区（extent）、页（page）/块（block）组成。

表空间

表空间——逻辑结构的最高层。不同的表空间使用一个唯一标识的space id 来标记。

区

区是由连续页组成的空间，任何情况下每个区的大小都为1MB。为保证区中页的连续性，InnoDB一次从磁盘申请4~5个区，InnoDB存储引擎默认页的大小为16KB，即一个区中一共有64个连续的页。

参数innodb_page_size，设置默认页的大小。页的大小发生变化，每个区对应页的数量发生变化，区大小固定不变。

在启用参数innodb_file_per_table后，创建的表默认大小是96KB。

区是64个连续的页，创建的表默认大小小于1MB的原因：
在每个段开始时，先用32个页大小的碎片页来存放数据，在使用完这些页之后才是64个连续页的申请。目的是：对于一些小表，或者是undo这类的段，可以在开始时申请较少的空间，节省磁盘容量的开销。

页

页——InnoDB磁盘管理的最小单位。

常见页类型：数据页、undo页、系统页、事务数据页、插入缓冲位图页、插入缓冲空闲列表页、未压缩的二进制大对象页、压缩的二进制大对象页。

行

InnoDB是面向行的（row-oriented），数据按行进行存放，每个页最多允许存放16KB/2-200行记录，即7992行记录。

InnoDB行记录格式

InnoDB 1.0.x之前，两种存放行记录数据的格式——Compact和Redundant。MySQL 5.1，默认设置为Compact行格式，通过show table status like ‘table_name’查看当前表使用的行格式（row_format属性表示当前所使用的行记录结构类型）。

InnoDB 1.0.x版本引入了新的文件格式（页格式），以前支持Compact和Redundant格式称为Antelope文件格式，新的文件格式称为Barracuda文件格式，拥有两种新的行记录格式：Compressed和Dynamic。

Compact行记录格式

  1. 变长字段长度列表，按照列的顺序逆序放置，其长度为：
    1. 若列的长度小于255字节，用1字节表示；
    2. 若大于255字节，用2字节表示。
  2. NULL标志位，指示该行数据中是否有NULL值，有则用1表示，该位占1字节。
  3. 记录头信息，固定占用5字节，共40位，每位含义见表4-1。

    每行数据除了用户定义的列之外，还有两个隐藏列，事务ID列（6字节）和回滚指针列（7字节），若InnoDB表没有定义主键，每行还会增加一个6字节的rowid列。

固定长度Char字段在未能完全占用其长度空间时，用0x20填充。

在compact格式下，NULL值不占用任何存储空间。

InnoDB在页内部通过一种链表的结构来串联各个行记录。

Redundant行记录格式

  1. 字段长度偏移列表，按照列的顺序逆序放置，若列的长度小于255字节，用1字节表示；若大于255字节，用2字节表示。
  2. 记录头信息，占用6字节（48位），每位含义见表4-2。n_fields、lbyte_offs_flags。

    Redundant行记录格式对于varchar类型的NULL值不占用任何存储空间，char类型的NULL值需要占用空间。

行溢出数据

InnoDB可以将一条记录中的某些数据存储在真正的数据页面之外——行溢出数据。一般情况下，InnoDB存储引擎的数据都是存放在页类型为B-tree node中，但是当发生行溢出时，数据的前768字节保存在数据页中，剩余部分的数据存放在Uncompressed BLOB Page中。

InnoDB存储引擎表是索引组织的，即B+Tree的结构，每个页中至少应该有两行记录，因此，如果页中只能存放下一条记录，InnoDB存储引擎会自定将行数据存放到溢出页中。

一个页中至少可以放入两行数据，varchar最大为8098。

对于任意类型的数据，只要至少保证一个页能存放两条记录，数据就存放在数据页中，而不会存放到Uncompressed BLOB Page中。

Compressed和Dynamic行记录格式


新的两种行记录格式对于存放在BLOB中的数据采用了完全的行溢出方式，在数据页中只存放20个字节的指针，实际的数据都存放在Off Page中。

对于Compressed行记录格式，存储在其中的行数据会以zlib算法进行压缩。

CHAR的行结构存储

如果字符集以单字节latin1格式为例，每行的变长字段长度列表不会存储char类型的长度。

对于多字节字符编码的char数据类型的存储，InnoDB存储引擎在内部将其视为变长字符类型，此时，在变长长度列表中会记录char数据类型的长度。因此可以认为在多字节字符集的情况下，char和varchar的实际行存储基本没有区别。

InnoDB数据页结构


记录在页中是根据堆的形式存放的。

File Header记录页的头信息；Page Header记录数据页的状态信息；Infimum & Supremum数据页中两个虚拟的行记录，用于限定数据的边界；Free Space是链表数据结构，当一条记录被删除后，该空间会被加入到空闲链表中。

Page Directory中存放记录的相对位置（页相对位置，而非偏移量），记录指针为Slots（槽）Directory Slots（目录槽）。InnoDB的槽是一个稀疏目录，即一个槽可能包含多个记录。当记录被插入或删除时需要对槽进行分裂或平衡的维护操作。在Slots中记录按索引键值顺序存放。

聚集索引和辅助索引，二者区别：叶子节点存放的是否是一整行的信息。

聚集索引按表的主键构造一棵B+树，叶子节点（数据页）中存放整张表的行记录数据，每个数据页都通过一个双向链表来进行链接。存储特点：逻辑上连续，物理上不连续。

辅助索引的叶子节点存放键值、书签（对应行数据的聚集索引键)。

当通过辅助索引来寻找数据时，InnoDB会遍历辅助索引并通过叶级别的指针获得指向主键索引的主键，然后再通过主键索引来找到一个完整的行记录。
